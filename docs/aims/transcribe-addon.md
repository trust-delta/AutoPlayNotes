# IS
重い機械学習依存（oemer / basic-pitch / 将来 opencv）を本体に同梱せず、隣接 dir の自己完結アドオンへ隔離する、という読み。契約は一貫して「アドオンが MIDI を吐き、本体が既存の `_load_score_file(midi)` で取り込む」＝本体側の配線を再利用し、エンジンを差し替え可能にする。同梱ビルドは `tools/build_*_addon.py`／`bundle_transcribe_addon.py` で 1 つの採譜アドオンへ束ねる（容量はローカル AI＝オフライン・無制限の付加価値として reframe）。

- OMR（`omr` ＋ oemer アドオン）: 綺麗な五線譜画像 → 譜面。実ビルド・E2E 済み。
- Phase 3a 音源→MIDI（`pitch` ＋ basic-pitch アドオン）: 音源のみの「弾いてみた」→ MIDI。実アドオン E2E 済み（合成音源は Score へ完全一致）。
- Phase 3b 映像→譜面（設計のみ）: Synthesia 系の落ちノーツ映像を semi-auto（本体トレースの 2 点キャリブ UX を流用）で譜面化。動画デコード（opencv）はアドオン側、キャリブは本体側という 2 フェーズ契約で、上記と同じ「アドオンが MIDI を吐く」形に帰着させる。

**品質軸は precision と音長の 2 つ**（2026-07-09 訂正）。旧前提「ペダルで音長がぼやけても、発音と音高が主役だから実害は小さい」は誤り。対象楽器は押している間だけ鳴るので、**音長はそのまま演奏の質になる**。basic-pitch がペダルで伸ばした音長を鵜呑みにすると、ゲーム内で音が濁る。緩和候補（音長のクオンタイズ / 最大音長のクリップ / ペダル区間の検出）は未着手。

**出力は下書き品質**であり、ユーザーが誤りを見つけて直す工程を必要とする。⚠️ **その工程が親 aim にまだ無い。** 「本体のエディタ / トレースで直す」という現在の前提は、譜面を読めるユーザーしか救わない。親の `[todo] 耳での検証導線` が入るまで、このアドオンの出力は**知識ゼロのユーザーには検証不能な形で出てくる**。素材の入口を広げるこの aim が、そのままスキルの壁を高くしている。
